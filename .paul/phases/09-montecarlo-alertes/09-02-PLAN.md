<objective>
## Goal
Système d'alertes in-app : règles de seuil par utilisateur, évaluées au refresh du portefeuille (`GET /api/assets`), qui génèrent des notifications dans la table `notifications` existante — avec dédup pour ne pas spammer.

## Purpose
Prévenir l'utilisateur quand un actif ou son portefeuille franchit un seuil (perte forte, gain fort). Réutilise le système de notifications déjà en place (GET /api/notifications). Backend seul — l'UI de config des seuils viendra ensuite.

## Output
- Table `alert_rules` (schéma dans database.py)
- `backend/services/alerts_service.py` : évaluation des règles → notifications dédupliquées
- Endpoints CRUD règles + hook d'évaluation dans le refresh portfolio
- Test pytest
</objective>

<acceptance_criteria>
## AC-1: CRUD des règles d'alerte
Given un utilisateur authentifié
When il POST une règle `{scope:"asset"|"portfolio", symbol?, direction:"below"|"above", metric:"day_change"|"vs_pru", threshold: number}`
Then la règle est persistée et listée par `GET /api/alerts`; DELETE la supprime.

## AC-2: Évaluation au refresh génère une notification
Given une règle « asset AAPL day_change below -5 » et AAPL à −6% aujourd'hui
When l'utilisateur appelle `GET /api/assets`
Then une notification est créée avec un message clair, visible dans `GET /api/notifications`.

## AC-3: Dédup — pas de spam
Given la même règle déjà déclenchée aujourd'hui (notification créée dans les dernières 24h pour cette règle)
When un nouveau `GET /api/assets`
Then AUCUNE nouvelle notification dupliquée n'est créée.

## AC-4: Validation & auth
Given input invalide (threshold non numérique, scope inconnu, direction hors below/above) ou pas de token
When appel des endpoints alertes
Then 422/400 (validation Pydantic) ou 401. Pas de crash.

## AC-5: Évaluation robuste
Given le service d'évaluation lève une erreur (données manquantes)
When `GET /api/assets` s'exécute
Then l'erreur est loggée/avalée proprement SANS casser la réponse assets (le refresh portfolio reste prioritaire).
</acceptance_criteria>
